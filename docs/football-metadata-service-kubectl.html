kubectl for football-metadata-service

A practical guide to inspecting and operating this service on EKS — what's deployed, how to observe it, how to run the ingestion job on demand, and what not to do.

GitOps first. ArgoCD is the source of truth: it renders this service's chart (football-metadata-service-chart) and continuously reconciles the cluster to match it. Treat kubectl as read & observe + run one-off jobs. Anything that changes desired state (kubectl edit, scale, apply, set image) will be reverted by ArgoCD — make those changes in the chart repo instead.

1. Prerequisites

You need cluster credentials (kubeconfig) built from an AWS role that can reach the EKS cluster (account 891239948857, us-east-1).

# one-time: point kubeconfig at the cluster (find the cluster name with `aws eks list-clusters`)
aws eks update-kubeconfig --name <cluster> --region us-east-1 --profile Production.Skcr-Software-Role

kubectl config current-context        # confirm you're on the right cluster
kubectl auth can-i get pods -n football-metadata-service-dev

2. What's deployed (mental model)

The chart wraps fastapi-base. Most resources come from that dependency; only two are defined in our chart.

Secrets are not Kubernetes secrets. The service's own DB creds are read at runtime via pires + IRSA from AWS (see app/core/settings.py), and the ingestion job reads wilson's DB via son (which fetches wilson's DB creds from AWS too). So kubectl get secret will not show them — check the ServiceAccount's eks.amazonaws.com/role-arn annotation and the AWS secret paths instead. The ingestion pod additionally needs network reach to wilson's Aurora and IAM to read wilson's DB-creds secret.

3. Everyday inspection

kubectl get all -n $NS                              # pods, svc, deploy, cronjob, jobs at a glance
kubectl get pods -n $NS -o wide                     # node, IP, restarts
kubectl describe pod <pod> -n $NS                    # events, env, mounts, why it won't start
kubectl logs -n $NS deploy/<app-deploy> -f           # follow API logs
kubectl logs -n $NS <pod> --previous                 # logs of the last crashed container
kubectl get events -n $NS --sort-by=.lastTimestamp  # recent cluster events (scheduling, pulls, OOM)
kubectl rollout status deploy/<app-deploy> -n $NS    # is the latest deploy healthy?

4. The ingestion CronJob (the useful part)

Runs daily — dev 03:00 UTC, prod 02:00 UTC. Command: python -m app.ingestion. It mirrors wilson players (create-only) and enriches the newly-created heimspiel-matched ones.

# find the exact (release-prefixed) name + schedule + last run
kubectl get cronjob -n $NS
CJ=$(kubectl get cronjob -n $NS -o name | grep ingest-players)

# trigger it NOW as a one-off Job (uses the exact cronjob spec: image, IRSA, env, resources)
kubectl create job -n $NS --from=$CJ ingest-players-manual-$(date +%s)

# watch it
kubectl get jobs,pods -n $NS | grep ingest-players
kubectl logs -n $NS job/ingest-players-manual-<ts> -f
Expected log tail: Ingest done: N created, M enriched, 0 failed. Because ingestion is create-only, re-running after players already exist prints created: 0 and enriches nothing new — that's normal, not a bug. The heavy run is the first one.
Most common failure: can't reach wilson's DB. Ingestion reads wilson via son, which bootstraps from SKILLCORNER_ENV and fetches wilson's DB creds from AWS. A boot/connect error (OperationalError, timeout, or a Pires/secrets error) means the pod lacks network reach to wilson's Aurora or IAM to read wilson's DB-creds secret. Fix the IRSA grant / networking, then re-trigger.

Pause / resume the schedule (temporary, will drift from Git)

kubectl patch $CJ -n $NS -p '{"spec":{"suspend":true}}'   # pause; ArgoCD may re-sync it back

For a durable change (schedule, suspend, command) edit environments/<env>/values.yaml in the chart repo instead.

5. Debugging a failing pod / job

kubectl describe pod <pod> -n $NS                    # Events section: ImagePull, OOMKilled, CrashLoop, probe fails
kubectl logs <pod> -n $NS --previous                 # the crash output
kubectl get pod <pod> -n $NS -o jsonpath='{.status.containerStatuses[0].state}{"\n"}'

6. Run a command in-cluster

# open a shell in the running API pod (shares its image, IRSA, env)
kubectl exec -it -n $NS deploy/<app-deploy> -- bash

# or run the ingestion directly in the app pod (alternative to the one-off Job)
kubectl exec -n $NS deploy/<app-deploy> -- python -m app.ingestion

The DB is Aurora (not in-cluster). To query it use the app's own scripts/psql_connect.py with the right AWS profile, or a bastion — not kubectl.

7. Hit the API / metrics locally

kubectl port-forward -n $NS svc/<app-svc> 8080:80

curl localhost:8080/api/docs        # OpenAPI UI
curl localhost:8080/metrics         # Prometheus (players_synced_total, players_enriched_total, …)
# /api/players requires a bearer JWT (see ADR on interim HS256 auth)

8. Safety notes

Never kubectl edit/apply/delete a managed resource to "fix" prod — ArgoCD reverts it and you've hidden the real change from Git. The only durable path is a chart-repo MR. Exceptions that are safe & ephemeral: creating a one-off Job --from=cronjob, reading logs, exec, port-forward, describe. Double-check $NS before every prod command.
