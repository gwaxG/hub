Claude Ops Hub

A project-agnostic assistant hub: memory · workflows · scheduling · phone.

This hub wires together capabilities Claude Code already ships instead of building a bespoke backend. Everything project-specific lives in one file — config/hub.config.yaml. Swap it to repurpose the hub.

Long-term & graph memory

Two layers with a clear split — don't duplicate between them:

LayerRoleBacked by
claude-memAutomatic "what I did" — captured & re-injected each sessionSQLite + FTS5 & local ChromaDB (all-MiniLM-L6-v2)
docs/Curated "what I've learned about the code" — a living, cross-linked Markdown knowledge base Claude maintainsplain Markdown: per-project notes in docs/workspace_graph/ + ADRs in docs/adr/
No extra database. claude-mem already does local keyword and semantic search. A separate sqlite-vec / vector MCP would duplicate it. If Chroma RAM gets heavy, switch claude-mem to SQLite-only mode.

The rule: read the relevant project note before exploring; update it (or write it, via /explain-repo) after learning something durable; record design decisions as ADRs. It's plain Markdown — readable on GitHub or in any editor, no app required.

Workflows & agents

  • Ad-hoc parallel work: the Agent tool with a fitting subagent type (Explore, Plan, code-reviewer).
  • Repeatable fan-out: Workflow scripts in workflows/, each reading config/hub.config.yaml.
  • Isolation: git worktrees (EnterWorktree, the using-git-worktrees skill, or isolation:"worktree" in a workflow) when agents edit in parallel.

Starter workflows: daily-brief (→ data/digest-<date>.html), incident-triage, mr-review.

Phone access

  • claude.ai app — authenticate connectors; digests & routine notifications land here.
  • Claude Code channels — enabled; to drive local sessions remotely set remote_control_at_startup: true in ~/.claude/policy-limits.json (confirm the in-app toggle first).
